# Definition of Done

## Purpose

This file defines when work can be considered complete.

## Universal Definition of Done

Work is done only when:

- Scope is clear.
- Requirements are satisfied.
- Acceptance criteria are addressed.
- Architecture impact is considered.
- Code is implemented if required.
- Tests are added or explicitly deemed unnecessary.
- Existing relevant tests pass or failures are documented.
- Code review is complete when code changed.
- Security risk is reviewed when relevant.
- Performance risk is reviewed when relevant.
- Documentation impact is addressed.
- CI/CD impact is addressed.
- Known risks are documented.
- No unresolved blockers remain.

## New Project Done

A new project planning phase is done when:

- PRD exists.
- User stories exist.
- Acceptance criteria exist.
- Architecture exists.
- Test strategy exists.
- Tasks exist.
- Risks exist.
- Implementation has not started without approval.

## Feature Done

A feature is done when:

- Approved scope is implemented.
- Acceptance criteria are met.
- Tests are added/updated or documented as not needed.
- Code review is complete.
- Documentation is updated if behavior changed.
- Known risks are documented.

## Bug Fix Done

A bug fix is done when:

- Root cause is documented.
- Fix is implemented.
- Regression risk is considered.
- Verification is performed.
- Regression coverage is added or documented as not needed.
- Review is complete.

## Test Work Done

Test work is done when:

- Tests map to requirements or risks.
- Test data is controlled.
- Tests are stable or flakiness is documented.
- Execution command is documented.
- Coverage gaps are visible.

## Release Done

A release is ready only when:

- Release checklist is complete.
- Quality gates are satisfied.
- Security review is complete when relevant.
- Known risks are documented.
- Rollback plan exists.
- Go/No-Go recommendation exists.
- Approval is received when required.
